## 一、全文总结（先读这里）

### 1.1 项目当前耦合核心问题

当前项目存在 **5 类核心耦合问题**：

| 问题 | 严重程度 | 影响范围 |
|------|---------|---------|
| ① `main.py` 是超级耦合单体 | 🔴 严重 | 35 处 import，23 个 slash 命令，CLI + 审批 + Agent 运行 + 错误处理全部混在一个文件 |
| ② `dynamic.py` 同时服务 serial/full 但没有区分 | 🔴 严重 | serial 和 full 调用同一个函数，仅靠 `execution_mode` 参数区分行为 |
| ③ `entry.py` 直接 import `main.py` | 🟡 重要 | CLI 外壳与内核无边界，改任何内核代码都可能影响 CLI 入口 |
| ④ 三种模式在 `chat_loop` 中 if/elif/else 硬编码 | 🟡 重要 | 新增模式需要改 main.py，无法独立部署/测试 |
| ⑤ 配置系统跨层调用 | 🟡 重要 | `model_catalog.py` 导入 `model_config.py`，`settings.py` 导入两者，形成网状依赖 |

### 3.3 dynamic.py — 串行/并行执行引擎不分（~1177 行）

**当前结构的核心问题**：

`execute_dynamic_request()` 是一个函数服务两种模式（serial 和 full），通过 `settings.execution_mode` 参数区分行为。但实际上，serial 和 full 的调用者（`run_serial_request` 和 `run_full_request`）代码完全一样：

```python
# runtime/modes/serial.py (28 行)
async def run_serial_request(...):
    return await execute_dynamic_request(...)

# runtime/modes/full.py (27 行)
async def run_full_request(...):
    return await execute_dynamic_request(...)
```

两个文件一模一样！真正的区分逻辑被推到 `dynamic.py` 内部的 `_execution_batches_for_mode()`：

```python
# dynamic.py 行 765-769
def _execution_batches_for_mode(tasks: list, execution_mode: str) -> list[list]:
    mode = normalize_execution_mode(execution_mode)
    if mode == "full":
        return _execution_batches_for_group(tasks)  # 可能并行
    return [[task] for task in tasks]                # 强制串行（每个任务单独一批）
```

**问题**：
1. serial.py 和 full.py 作为"模式入口"实际上什么都没做——它们只是传声筒
2. 真正区分模式行为的代码埋在 dynamic.py 内部深处
3. 如果 serial 模式需要一个完全不同的执行策略（比如不同的错误处理），必须改 dynamic.py
4. 这违反了"模式应该独立演化"的原则

**dynamic.py 内部的耦合职能**：

```
execute_dynamic_request() 承担的职责（应拆分为多个独立模块）：
├─ ① 修复循环控制          — for attempt in range(1, max_attempts+1)
├─ ② 规划管线调用          — preview_plan() 
├─ ③ Gate 决策             — apply_pipeline_gate()
├─ ④ 计划校验              — validate_plan()
├─ ⑤ 计划审查              — review_plan()

├─ ⑦ route_type 分发       — direct_answer / clarify / single_agent / multi_agent
├─ ⑧ 快速路径检测          — _can_fast_path_url_search, _can_fast_path_git_status
├─ ⑨ 内联文件上下文        — _inline_project_file_context()
├─ ⑩ 单 Agent 执行         — AgentFactory + run_agent
├─ ⑪ 多 Agent 并行调度     — _run_multi_agent()
```

## 四、三大核心耦合问题深挖

### 4.1 问题①：三种模式没有真正的边界

**现象**：
- `serial.py` 和 `full.py` 代码完全相同（27-28 行各）
- 真正的区分逻辑在 `dynamic.py` 内部的 `_execution_batches_for_mode()`
- solo 模式完全不经过 `dynamic.py`，而是直接走 `solo.py → AgentFactory.create_solo_agent() → run_agent()`

**根本原因**：三种模式本来是三种不同的"执行策略"，但它们共享了同一个 `execute_dynamic_request()` 实现，导致：
1. 模式间没有明确接口
2. 新增模式特性必须在 dynamic.py 内部添加条件分支
3. 无法对单个模式做性能优化

**正确做法**：每种模式应该是一个独立的"执行策略"，实现统一的 ExecutionStrategy 接口。

### 4.2 问题②：main.py 集成了太多不同的子系统

**现象**：main.py 的 `chat_loop()` 函数同时处理：
- Slash 命令路由（23 个命令）
- 三种模式的选择和执行
- MCP Server 生命周期
- 会话检查点管理
- 上下文管理（recent_turns）

**根本原因**：没有"请求处理管线"的抽象。chat_loop 是直接操纵所有子系统。

**正确做法**：引入 Kernel 门面，chat_loop 只负责"读取输入 → 交给 Kernel → 输出结果"。

### 4.3 问题③：CLI 和内核没有分层

**现象**：
- `entry.py` 直接 import `main.py`
- `entry.py` 的 `_handle_run()` 直接创建 ModelRegistry、MCPServerManager
- CLI 命令处理函数（_handle_connect、_handle_models 等）直接调用 config 子系统

**根本原因**：没有定义内核的对外接口。CLI 代码必须知道内核的每个细节才能工作。

**正确做法**：定义 Kernel Facade，CLI 只依赖 Facade 接口。

## 五、解耦设计方案

### 5.1 设计原则

1. **依赖倒置**：高层模块（CLI）不应该依赖低层模块（Kernel 内部实现），两者都应该依赖抽象接口
2. **接口隔离**：每种模式有独立的 ExecutionStrategy，每个子系统有独立的 Facade
3. **公共池子不动**：catalog_system、runtime/config、runtime/common、skills 作为共享基础设施保持不变
4. **渐进式**：每次只迁移一个子系统，保持项目始终可运行

### 5.2 核心改造①：Kernel Facade — CLI 与内核的分层边界

**新建文件**：`runtime/kernel/__init__.py`

```python
"""Kernel Facade — CLI 外壳与内核的唯一接触点。

CLI 层（entry.py, main.py 的 chat_loop）只需要知道这个模块的接口。
内核内部的任何改动，只要接口不变，CLI 就不会受影响。
"""

from dataclasses import dataclass
from pathlib import Path
from typing import Protocol, Callable, Awaitable


# ── 接口定义 ──────────────────────────────────────────────

@dataclass
class KernelRequest:
    """一次用户请求的标准化输入（CLI → Kernel）"""
    user_input: str
    workspace_root: Path
    execution_mode: str           # "solo" | "serial" | "full"
    recent_context: str = ""      # 压缩后的历史上下文


@dataclass  
class KernelResponse:
    """一次请求的标准化输出（Kernel → CLI）"""
    final_output: str
    stopped: bool = False
    token_summary: dict | None = None
    mcp_ids_used: list[str] | None = None
    turn_status: str = "完成"     # "完成" | "失败" | "已中断" | "待修复"


class KernelSession(Protocol):
    """内核会话接口（CLI 只依赖此接口）"""
    
    async def handle_request(self, request: KernelRequest) -> KernelResponse:
        """处理一次用户请求，返回标准化响应"""
        ...


class ExecutionStrategy(Protocol):
    """执行模式策略接口（solo/serial/full 各自实现）"""
    
    @property
    def mode_name(self) -> str: ...
    
    async def execute(
        self,
        request: KernelRequest,
        model_registry,
        mcp_manager,
        hooks,
        run_agent,
        checkpoint_manager,
        settings,
    ) -> KernelResponse: ...
```

**这一步的价值**：
- CLI 不再知道内核内部有多少子系统
- 内核内部可以自由重构（拆/合/替换模块），CLI 完全无感
- 可以独立测试 CLI 和内核

**策略工厂**：

```python
# runtime/kernel/strategies/__init__.py
def create_execution_strategy(mode: str) -> BaseExecutionStrategy:
    strategies = {
        "solo": SoloStrategy(),
        "serial": SerialStrategy(), 
        "full": FullStrategy(),
    }
    strategy = strategies.get(mode)
    if strategy is None:
        raise ValueError(f"Unknown execution mode: {mode}")
    return strategy
```

**这一步的价值**：
- 每种模式是独立类，可以独立开发、测试、部署

- 模式之间可以有不同的依赖（solo 不需要 planning，serial/full 需要）
- serial.py 和 full.py 的当前"空壳"文件可以删除

## 八、回审与优化

### 8.1 回审点 1：解耦后是否引入了新的复杂度？

**可能的担忧**：拆分成更多文件后，新人更难理解项目。

**缓解措施**：
1. 每个新文件以清晰的文件名描述职责（如 `parallel_scheduler.py`）
2. Kernel Facade 的 5 个接口作为"项目地图入口"——新人只需看懂这 5 个接口就能理解整个系统
3. 当前 965 行的 main.py 和 1177 行的 dynamic.py 才是真正的理解障碍

**结论**：拆分降低而非增加理解成本。

### 8.2 回审点 2：是否有过度设计？

**可能的担忧**：ExecutionStrategy Protocol 和 Kernel Facade 是过度抽象。

**自我审查**：
- ExecutionStrategy：三种模式确实有不同的行为，策略模式正好匹配
- Kernel Facade：当前 entry.py 直接 import main.py 是真实的痛点，门面模式是标准解

**不是过度设计**。如果项目只有一种执行模式，策略模式确实是过度设计——但项目明确有三种模式且可能增加第四种。

### 8.3 回审点 3：是否遗漏了重要的耦合？

**检查**：
- [x] entry.py ↔ main.py：已覆盖（阶段 4）
- [x] main.py 内部职责混合：已覆盖（阶段 1、2、4）
- [x] serial/full 代码相同但行为不同：已覆盖（阶段 2）
- [x] dynamic.py 职责过多：已覆盖（阶段 3）
- [x] skills ↔ planning 循环依赖：已覆盖（阶段 1 微调）
- [x] settings.py ↔ model_catalog.py 反向依赖：已覆盖（第六章微调）
- [x] planner_schema.py 中的业务逻辑：已覆盖（阶段 1 微调，移到 agent 层）
- [ ] catalog_system 内部的 model_catalog.py 行数过多（592 行）→ 可以后续再拆，不算紧急

### 8.4 回审点 4：实施顺序是否合理？

当前顺序：
1. 阶段 1（微创手术）：最小改动，建立信心 ✅
2. 阶段 2（模式策略化）：核心架构调整 ✅

顺序合理。每个阶段都可以独立合并和测试，不会破坏后续阶段。

> **全文结论**：当前项目的主要耦合集中在 `main.py`（超级单体）、`dynamic.py`（职责过多）、以及 CLI 与内核的无边界交互。通过 4 个阶段的渐进式解耦——微创手术 → 模式策略化 → 执行引擎拆分 → Kernel Facade——可以在不破坏任何现有功能的前提下，实现 CLI 外壳与 Kernel 内核的完全分层、三种执行模式的独立演化、以及每个功能模块的独立部署和测试。公共池子（catalog_system、runtime/config、skills 等）保持不动，确保改动的影响范围可控。
