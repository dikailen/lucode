# Lucode 产品化路线图

> 日期：2026-05-12
> 目标：从当前原型状态演进到可对标 Claude Code / OpenCode 的终端编码 Agent 产品
> 参考：解耦评审报告 + 架构差距分析 + Claude Code/OpenCode 产品特性

---

## 〇、当前状态定位

Lucode 处在 **"技术验证完成，工程化不足"** 阶段：

- ✅ 核心链路走通：用户输入 → 规划管线 → 多 Agent 执行 → 审计反馈
- ✅ MCP 工具层较完整：7 个 Server，安全分层，子进程隔离
- ✅ 三种执行模式：solo / serial / full
- ✅ 安全基础：隐私模式、审计引擎、回滚检查点
- ⚠️ 架构解耦完成 55%，`dynamic.py` 仍是 1176 行单体
- ❌ 无产品化包装：无 installer、无 CI 发布、无版本管理
- ❌ 终端 UI 为纯 print 文本
- ❌ 仅支持 OpenAI-compatible 协议

## 一、产品化五阶段总览

```
阶段1: 工程补课 (2周)    → 先把代码质量提到可维护水平

阶段5: 发布运维 (持续)    → 打包分发 + 文档 + 社区
```

## 三、阶段2：核心体验（Week 3-6）

> 目标：用户第一次用就感到"这玩意能用"，而非"这还在实验阶段"。

### 3.1 启动速度（感知第一要素）

```
当前: lucode --version → 1.5s+ (完整Python import链)
目标: lucode --version → <50ms (零导入)
      lucode chat    → <500ms (并行预取，先渲染再加载)
```

| 子任务 | 优先级 | 说明 |
|--------|--------|------|
| 快速路径分发：`--version` / `--help` 零导入 | P0 | entry.py 手工解析 argv，命中快速路径直接 return |
| 早期输入捕获线程 | P1 | import main.py 期间启动 stdin 读取线程，防按键丢失 |
| 启动阶段并行化 | P1 | `refresh_catalogs()` 改为后台任务，不等它完成就开始 REPL |
| 懒 import 化 | P1 | main.py 顶层 import 改为函数内按需 import |

**验收标准：** 在 5 年老笔记本上，`lucode --version` < 100ms，`lucode chat` 到出现 prompt < 800ms。

### 3.2 终端 UI 升级（从 print 到 TUI）

这是与 Claude Code 差距最大的用户感知维度。当前纯 print 输出，无结构化渲染。

**技术选型建议：** 使用 `rich` 库（Python 生态最成熟的终端渲染库），而非从零造轮子。

| 子任务 | 说明 |
|--------|------|
| **Status Line** | 底部固定行，显示：当前模式、活跃 MCP、模型名、本轮 token 用量 |
| **Spinner 状态机** | thinking → planning → executing → reviewing，每阶段有独立动画 |
| **工具调用卡片** | 折叠显示：工具名 + 参数摘要 + 结果行数，可展开看完整结果 |
| **Diff 渲染** | 利用 rich 的 `Syntax` 组件做语法高亮 diff，替代当前纯文本 |
| **审批弹窗** | 用 rich 的 `Panel` 渲染工具预览、参数格式化，取代当前 print |
| **Markdown 流式渲染** | 模型输出的 Markdown 增量解析渲染，支持代码块高亮 |
| **/model 选择菜单** | 用 `questionary` 或 rich 的 `Prompt` 做交互式模型切换 |

**验收标准：** 截图可以放到 README 里当宣传素材。

### 3.3 错误处理产品化

当前错误信息是开发者导向的技术错误，用户看到 `APIConnectionError` 会困惑。

| 子任务 | 说明 |
|--------|------|
| 错误分级：致命 / 可恢复 / 警告 | 致命退出，可恢复友好提示，警告静默 |
| 网络错误 → "模型 API 连接失败，请检查网络和 API Key。稍后重试。" |
| 超时错误 → "模型响应超时。已自动重试（1/3）..." |
| 配额错误 → "API 额度已用完。建议切换 fallback 模型或检查账户余额。" |
| 模型不支持工具 → "当前模型不支持工具调用，已自动切换为纯文本对话模式。" |
| 工具调用失败 → 不崩溃，将错误作为 ToolResult 注入消息流，让 LLM 自适应 |

## 四、阶段3：多厂商模型支持（Week 7-10）

> 目标：打破 "只能 OpenAI-compatible" 的限制，成为真正的多模型终端 Agent。

### 4.1 架构：Provider → Adapter → 统一接口

```
                ┌──────────────────────────────┐
                │     Agent Loop (统一接口)      │
                │  send_message() → stream/非流  │
                └──────────┬───────────────────┘
                           │
                ┌──────────▼───────────────────┐
                │    MessageTransformer        │
                │  按 provider 类型清洗消息      │
                └──────────┬───────────────────┘
                           │
          ┌────────────────┼────────────────┐
          ▼                ▼                ▼
   ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
   │ Anthropic    │ │ OpenAI       │ │ Google       │
   │ Adapter      │ │ Adapter      │ │ Adapter      │
   │ (原生SDK)    │ │ (原生SDK)    │ │ (原生SDK)    │
   └──────────────┘ └──────────────┘ └──────────────┘
          │                │                │
          └────────────────┼────────────────┘
                           ▼
                ┌──────────────────────┐
                │  OpenAI-compatible   │
                │  (通用 fallback)      │
                └──────────────────────┘
```

### 4.2 优先级排序（按用户覆盖率）

| Provider | 优先级 | 用户场景 | 技术要点 |
|----------|--------|---------|---------|
| **Anthropic** | P0 | Claude 用户（最大 AI 编码用户群） | 原生 SDK、thinking、prompt caching、tool use 协议 |
| **OpenAI 原生** | P1 | ChatGPT/Codex 用户 | 原生 SDK、reasoning_effort、structured outputs |
| **Google Gemini** | P1 | 免费额度用户 | 原生 SDK、长上下文、多模态 |
| **OpenRouter** | P2 | 一个 API Key 用所有模型 | OpenAI-compatible 路径 |
| **Ollama** | P2 | 完全本地运行 | 本地推理、隐私模式 |

### 4.3 单项深度拆解：以 Anthropic Adapter 为例

因为 Anthropic 的 tool use 协议与 OpenAI 有显著差异，需要重点处理：

| 差异点 | OpenAI | Anthropic | 适配方案 |
|--------|--------|-----------|---------|
| tool_call ID | `call_xxx` | `toolu_xxx` | 规范化映射 |
| 空 content | API 接受 | 400 Bad Request | transform 层替换为 `"(empty)"` |
| system 消息 | 顶层 `system` 角色 | 独立 `system` 参数 | adapter 层分离 |
| thinking | 无原生支持 | `thinking.budget_tokens` | 暴露到配置层 |
| prompt caching | 无原生支持 | `cache_control` breakout | 高频 system/tools 块标记 |
| 流式 tool_use | `delta.tool_calls` | `content_block_start/delta/stop` | adapter 层统一为事件流 |

### 4.4 结构化配置系统

从平铺 `.env` 迁移到 `lucode.json` 结构化配置（兼容 `.env` 读取）：

```json
{
  "provider": {
    "deepseek": {
      "type": "openai_compatible",
      "base_url": "https://api.deepseek.com",
      "api_key_env": "DEEPSEEK_API_KEY",
      "models": {
        "deepseek-chat": { "max_tokens": 8192 },
        "deepseek-reasoner": { "reasoning_effort": "high" }
      }
    },
    "claude": {
      "type": "anthropic",
      "api_key_env": "ANTHROPIC_API_KEY",
      "models": {
        "claude-sonnet-4-6": {
          "thinking": { "type": "enabled", "budget_tokens": 16000 },
          "max_tokens": 8192
        }
      }
    }
  },
  "default_model": "deepseek/deepseek-chat",
  "fallback_models": ["claude/claude-haiku-4-5"],
  "mode": "full",
  "privacy": "local_first"
}
```

## 五、阶段4：智能深度（Week 11-16）

> 目标：长对话不退化、权限可扩展、执行策略可控。

### 5.1 Context 压缩管线

这是决定"一次对话能走多远"的核心能力。

```
触发条件: 上下文利用率 > 83%
  ↓
Level 1 - TRUNCATE (压缩到 80%): 裁剪大型工具输出，保留头尾
  ↓ 仍超限
Level 2 - DEDUPLICATE (压缩到 85%): 合并重复文件内容引用
  ↓ 仍超限
Level 3 - FOLD (压缩到 90%): 折叠已完成任务的中间步骤
  ↓ 仍超限  
Level 4 - SUMMARIZE (压缩到 95%): 用小模型生成结构化摘要
  └─ 保留: 用户意图 + 技术决策 + 已修改文件列表 + 剩余任务 + 错误记录
```

压缩后自动恢复上下文：
- 最近编辑的 5 个文件内容（头 500 行）
- 活跃 Skill 的系统提示
- 当前 workspace 目录树

### 5.2 多 Agent 协作增强

当前已有 sub-agent 模式（通过 SDK handoff），需要增强：

| 模式 | 当前状态 | 产品化目标 |
|------|---------|-----------|
| Sub-Agent | ✅ 已实现 | 允许用户指定子 Agent 的模型（如用 haiku 处理简单搜索） |
| Coordinator | ⚠️ 部分具备 | Orchestrator 只读不写，明确分工边界 |
| Swarm | ❌ 无 | 对等 Agent 隔离 worktree，互不干扰，最后合并结果 |
| 人机协作 | ❌ 无 | Agent 可主动向用户提问澄清需求，而非盲目执行 |

### 5.3 Hooks 系统

对标 Claude Code 的 hooks，让高级用户和团队可以自定义行为：

```python
# .lucode/hooks.toml
[hooks.PreToolUse]
# 只读文件操作，自动允许
auto_allow_read = { tools = ["read_file", "list_directory", "glob", "grep"] }

# 写入操作，执行前运行检查脚本
pre_write_check = { tools = ["write_file", "replace_in_file"], command = "check_file_size.sh" }

# Git 提交前自动运行测试
pre_commit = { tools = ["git_commit"], command = "pytest --quiet" }

[hooks.PostToolUse]
# 每次写入后自动格式化
auto_format = { tools = ["write_file", "replace_in_file"], command = "ruff format $FILE_PATH" }

[hooks.SessionStart]
# 启动时加载开发环境
load_env = { command = "source venv/bin/activate && echo 'venv activated'" }
```

事件类型：`PreToolUse` / `PostToolUse` / `SessionStart` / `SessionEnd` / `PreCompaction` / `PreMessage`

退出码协议：0=允许, 2=阻止并告知用户, 其他=允许但记录警告日志

### 5.4 会话持久化

| 能力 | 说明 |
|------|------|
| **JSONL 消息存储** | 每轮对话追加写入 `.lucode/sessions/<id>.jsonl` |
| **`/resume` 恢复** | 列出历史会话，选择恢复，重建消息上下文 |
| **`/fork` 分支** | 从某个会话点复制状态，开始新分支（适合"A/B 试方案"） |
| **会话元数据** | 记录：起止时间、模型、token 总量、修改文件列表、任务摘要 |
| **自动清理** | 超过 30 天的会话自动归档 |

## 七、差异化定位建议

与 Claude Code 和 OpenCode 相比，Lucode 的差异化空间在哪？

| 维度 | Claude Code | OpenCode | Lucode 机会点 |
|------|-------------|----------|-------------|
| 生态绑定 | Anthropic 优先 | Vercel/AI SDK 生态 | **完全中立**，任一模型任一厂商 |
| 部署方式 | SaaS + CLI | CLI | **嵌入式**（可作为库集成到其他 Python 应用） |
| 会话模型 | 单 Agent | 单 Agent | **原生多 Agent** 规划-执行-审计管线 |
| 中文支持 | 有 | 有限 | **中文优先**（提示词、错误信息、文档均中文原生） |
| 协议 | 私有 | MCP | **MCP 原生**，工具层全部 MCP |
| 隐私 | 依赖 Anthropic | 依赖所选厂商 | **本地优先**架构，可完全离线运行（Ollama） |

**核心定位：模型中立 + 中文原生 + 本地优先的开源终端编码 Agent。**

## 八、关键决策点

### 决策1：Agent Loop — 继续委托 SDK 还是自研？

| 方案 | 优势 | 劣势 |
|------|------|------|
| 委托 SDK（现状） | 维护成本低、跟随上游演进 | 无法控制单次 turn 内的策略 |
| 自研 Agent Loop | 完全控制、可定制化 | 维护成本高、需要自己处理所有边界情况 |

**建议：** 阶段1-3 保持委托 SDK，阶段4 评估是否自研。如果用户对"工具调用可控性"需求强烈（如：一定要先读后写、写入前必须 diff），则自研。

### 决策2：终端 UI — rich 库还是自研？

**建议：** 用 `rich`。它是 Python 生态最成熟的 TUI 库，Textual 作者维护，被 pip、Poetry 等项目使用。自研 ANSI 差分引擎的成本极高且非核心壁垒。

### 决策3：配置格式 — TOML 还是 JSON？

**建议：** `.lucode/config.toml`（已有雏形）+ `lucode.json`（结构化 Provider 配置）。TOML 适合人类书写（注释、简洁语法），JSON 适合程序生成和嵌套结构。两者共存，TOML 覆盖用户配置，JSON 覆盖 Provider 声明。

### 决策4：Python 版本要求

**建议：** 要求 Python ≥ 3.11。理由：`asyncio.TaskGroup`（3.11）、`tomllib`（3.11 标准库）、更好的异常追踪。放弃 3.10 及以下以降低维护负担。

## 十、总结

Lucode 的核心资产——Planning 管线、多 Agent 执行、MCP 工具层、安全审计——已经有了不错的骨架。产品化要解决的是"让这骨架有血有肉，让用户愿意用、用了不后悔"。

**优先级一句话：先让启动快起来（1天），再让终端好看（2周），然后支持 Claude 模型（1周），这三件事做完，产品就初步"可以给人看"了。**
