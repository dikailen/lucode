## 二、做得好的（超出方案预期）

### 1. main.py 瘦身：965行 → 149行（方案目标 ~200行）

最亮眼的成果。`main.py` 从"项目最复杂的文件"变成了纯粹的启动入口 + 薄代理函数层。方案规划分4步迁移，最终效果比预期还好。

### 2. Shell 层目录结构比方案更完善

方案建议5个文件，实际建了5个且职责划分清晰：

```
lucode/shell/
├── chat_loop.py       ← 交互 REPL 主循环
├── slash_commands.py  ← 23 个 slash 命令分发
├── input_adapter.py   ← stdin 适配 + 运行时命令会话
├── turn_display.py    ← 错误格式化 + 状态显示
└── runtime_env.py     ← 环境变量集中管理（超出方案的增量）
```

`runtime_env.py` 是超出方案的增量，把散落各处的环境变量读取集中管理。

### 3. KernelFacade 接口设计比方案更干净

方案定义的 `KernelSession Protocol` 被更简洁的 `KernelFacade` 类替代，`KernelRequest`/`KernelResponse` 用 dataclass 实现，`print_summary` 通过闭包注入而非额外字段。实际设计优于方案设计。

### 4. `_handle_run()` 瘦身极成功

从方案列出的 9个 import + ~55行，缩减到 2个 import + ~10行：

```python
# entry.py L201-212
async def _handle_run(args, context) -> int:
    from runtime.kernel import KernelFacade
    ...
    response = await KernelFacade(context).run_once(prompt, show_plan=True)
```

### 5. 循环依赖修复

`skills/loader.py` → `planning/planner_schema.py` 的循环依赖已通过 `runtime/common/text_utils.py` 消除。`skills/loader.py` 现在从 `runtime/common/text_utils` 导入 `sanitize_text`。

## 三、未完成的硬伤

### 🔴 硬伤1：`dynamic.py` 完全没有拆分（方案阶段3 = 0%）

方案明确要求将 `dynamic.py` 的 17 种职责拆分到 7 个独立模块，目标从 1177 行缩减到 ~200 行编排器。

**现状：`dynamic.py` 仍是 1176 行，内部结构未动。**

未创建的文件：
- `runtime/execution/orchestrator.py`
- `runtime/execution/task_runner.py`
- `runtime/execution/parallel_scheduler.py`
- `runtime/execution/fast_paths.py`
- `runtime/execution/inline_context.py`
- `runtime/execution/workspace_helpers.py`

**影响**：这是方案中最大的单次代码迁移（阶段3，2-3天工作量），跳过它意味着核心执行引擎的耦合未解除。新增执行策略、修改错误处理、优化并行调度仍需在 1176 行的巨型函数中操作。

### 🔴 硬伤2：`runtime/modes/` 僵尸层未删除

方案明确要求"阶段2完成后删除 `runtime/modes/` 整个目录"。现状形成**三层委托反模式**：

```
策略层（策略模式）       → 门面层（空壳委托）         → 实际逻辑
SoloStrategy (15行)    → runtime/modes/solo.py     → solo.py 包含业务逻辑
SerialStrategy (20行)  → runtime/modes/serial.py   → dynamic.py (1176行)
FullStrategy (20行)    → runtime/modes/full.py     → dynamic.py (同上)
```

- `serial.py` 和 `full.py` 仍是 27 行完全相同的传声筒（方案已明确指出此问题）
- 现在只是外层加了一层策略壳，核心问题未解决

连测试都期望不应该再有这个依赖：
```python
# tests/run_regression.py L1799
self.assertNotIn("from runtime.modes", source)
```

### 🟡 硬伤3：Strategies 是壳，业务逻辑在别处

方案要求 strategies **包含**模式差异化逻辑。实际三个 Strategy 类各 10-20 行，只做参数转发：

| 策略类 | 实际代码行数 | 包含业务逻辑？ | 逻辑实际在哪？ |
|--------|------------|-------------|-------------|
| SoloStrategy | 15行 | 否 | `runtime/modes/solo.py` |
| SerialStrategy | 20行 | 否 | `runtime/execution/dynamic.py` |
| FullStrategy | 20行 | 否 | `runtime/execution/dynamic.py` |

方案的目标是 strategies 各自独立演化，但现状下改 serial 行为仍需改 `dynamic.py` 内部的条件分支。

### 🟡 硬伤4：`_normalize_model_reference` 仍在 `planner_schema.py`

方案 8.3 节明确标记为"已覆盖"的项，但该函数仍在 `planner_schema.py:463`，内部懒加载 `catalog_system.model_catalog` 和 `runtime.safety.privacy`。数据结构文件包含业务逻辑的反模式未修复。

### 🟡 硬伤5：`settings.py` → `model_catalog.py` 反向依赖未解

`_dynamic_default_priorities()` 仍在 `settings.py:129`，导致配置层反向依赖目录层。方案建议移到 `model_catalog.py` 或新建 `runtime/config/defaults.py`，未执行。

## 五、额外发现的改进点（方案未覆盖）

### 1. `main.py` 代理函数层（L41-104）是过渡方案

`create_token_logger_hooks`、`_approval_prompt`、`_run_agent_once` 等代理函数用懒加载避免循环 import。方案预期这些函数直接在新模块中使用。可工作的过渡方案，但增加了理解成本，后续应清理。

### 2. `runtime/execution/__init__.py` 为空（1字节）

执行引擎没有模块级接口，所有消费者直接导入 `dynamic.py`。缺少 `__all__` 导出或 facade。

### 3. `entry.py` 其他 handler 未使用 KernelFacade

`_handle_doctor`、`_handle_readonly`、`_handle_connect`、`_handle_models`、`_handle_auth` 等仍直接依赖 `catalog_system`、`runtime.config.*` 等内部子系统。方案 4.2 的改造只做了 `_handle_run`。

### 4. `main.py` 第 141-144 行冗余别名

```python
async def chat_loop(*args, **kwargs):
    from lucode.shell.chat_loop import chat_loop as shell_chat_loop
    return await shell_chat_loop(*args, **kwargs)
```

这个别名函数未被使用，可以删除。

## 七、结论

**项目确实进行了实质性解耦，但只走了方案的约 55%。**

走得最好的是 CLI 外壳与内核的分离（阶段1+4），触达了"最痛"的 `main.py`（从超级单体变为 149 行）和 `entry.py`（_handle_run 从 55 行缩至 10 行）。

走得最差的是执行引擎内部（阶段3），`dynamic.py` 纹丝未动，仍是 1176 行怪兽。`runtime/modes/` 目录变成了策略层和实际逻辑之间的僵尸代理层。

**形象评价：外壳手术做得很漂亮，内脏手术还没动刀。**

### 建议优先完成

1. **将 `runtime/modes/` 的逻辑合并入 strategies，删除 modes 目录**（1天，消除三层委托反模式）
2. **启动 `dynamic.py` 拆分**（2-3天，方案阶段3全部内容，最大收益）
3. **清理 `planner_schema.py` 中的 `_normalize_model_reference`**（0.5天，微调）
4. **清理 `main.py` 中的冗余代理函数和别名**（0.5天，最终清理）
