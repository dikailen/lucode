# 后续改进计划1期

## 核心方向

Lucode 后续不只是追赶 Claude Code / OpenCode，而是形成自己的产品方向：

```text
中文优先 + 多脑可配置 + CLI 优先 + MCP 兜底 + 可视化终端 Agent
```

核心原则：

- 三脑/多脑是 Lucode 的特色，要明确暴露给用户。
- 用户可以随时按自己的偏好切换每个脑的模型。
- 高频确定性任务优先走 CLI / native fast path。
- MCP 用于外部生态、远程工具和兜底。
- 执行过程要动态展示“现在正在做什么”。

## 1. 多脑模型调音台

用户既可以统一切换模型，也可以单独切换每个脑的模型。

### 当前状态

```text
状态：核心完成，可用状态
已完成：
- /models 独立多脑模型调音台。
- 上下键/鼠标候选选择，q 退出。
- /models select 统一默认模型。
- /models brain <脑位> <provider/model> 单独切换四脑模型。
- /models available 查看当前可运行模型。
- /models brain reset 重置项目多脑覆盖配置。
- 当前项目配置采用 [roles] 结构，运行时代码已围绕该结构稳定工作。
```

### 命令设计

```text
/models
/models available
/models select <provider/model>

/models brain
/models brain refiner <provider/model>
/models brain planner <provider/model>
/models brain executor <provider/model>
/models brain synthesizer <provider/model>
/models brain reset
```

支持中文别名：

```text
/models brain 前置优化 qwen/qwen-turbo
/models brain 主脑 qwen/qwen-max
/models brain 执行 deepseek/deepseek-chat
/models brain 汇总 kimi/k2
```

### 展示效果

```text
模型配置

默认模型：
- deepseek/deepseek-chat

多脑模型：
- 前置优化脑：qwen/qwen-turbo
- 主脑规划脑：qwen/qwen-max
- 执行专家脑：mimo/mimo-v25-pro
- 汇总脑：deepseek/deepseek-chat

当前模式：
- solo：使用执行专家脑
- serial/full：前置优化脑 -> 主脑规划脑 -> 执行专家脑 -> 汇总脑
```

### 配置结构

当前实现采用更贴近运行时代码的 `[roles]` 结构，不再使用早期草案里的 `[models.brains.*]` 嵌套结构。短期不建议迁移，避免破坏现有 `/models`、`RuntimeSettings` 和执行链。

```toml
[model]
primary = "deepseek/deepseek-chat"
fallback = ["qwen/qwen-plus"]

[roles]
query_refiner = ["qwen/qwen-turbo"]
orchestrator = ["qwen/qwen-max"]
executor = ["mimo/mimo-v25-pro"]
final_synthesizer = ["deepseek/deepseek-chat"]
```

## 2. Provider / API Key 体验

目标是不让用户手动改 `.env`。

### 第二阶段详细计划表

| 步骤 | 目标 | 主要改动位置 | 具体动作 | 验收标准 |
|---|---|---|---|---|
| 2.1 现状审计 | 确认 Provider、auth、catalog、CLI 的真实边界 | `runtime/config/model_config.py`、`catalog_system/model_catalog.py`、`lucode/entry.py`、`runtime/config/cli.py` | 梳理当前 `connect_provider`、`auth.json`、`.lucode/config.toml`、Provider 预设和模型 catalog 合并路径；列出不改代码也能跑通的命令 | 形成当前行为清单：key 存哪里、base_url 存哪里、模型怎么进 catalog、哪些命令能展示 |
| 2.2 Provider 预设补齐 | 让用户选择常见厂商时不必查 base_url | `catalogs/provider_catalog.json`、Provider 相关测试 | 检查并补齐 DeepSeek、OpenAI、OpenRouter、DashScope、SiliconFlow 的 `homepage`、`base_url`、`models`、`supports_tools`、`compatible_type` | `/connect <provider>` 能展示官网、请求地址、推荐模型；缺 key 时不泄露也不报 traceback |
| 2.3 `/connect` 命令收口 | 让终端命令真正成为配置入口 | `lucode/entry.py`、`runtime/config/cli.py`、`runtime/config/model_config.py` | 保持 `lucode connect provider --api-key` 可用；补 `/connect provider` 的只读引导；自定义中转必须校验 `homepage/base_url/model/api_key` | 新用户只输入一条 connect 命令即可写入 `auth.json` 和 `.lucode/config.toml` |
| 2.4 密钥存储边界 | API key 只进用户级 auth，不进项目配置 | `runtime/config/model_config.py`、`tests/run_regression.py` | 加强 `save_auth` 权限与脱敏展示；确认 `.lucode/config.toml` 永不写明文 key；`/api show` 和 `/connect` 输出统一隐藏 key | 测试里搜索输出和项目配置，不能出现 `sk-` 明文 |
| 2.5 自定义中转体验 | 支持用户自己的代理或中转站 | `connect_provider`、CLI 参数解析、Provider catalog 合并 | 允许 `lucode connect my_proxy --custom --homepage ... --base-url ... --model ... --api-key ...`；模型引用统一为 `my_proxy/model_name` | 自定义 provider 出现在 `/models` 可选模型中，并能被 `/models brain 执行 my_proxy/model` 选择 |
| 2.6 `/models list` 体验 | 用户能看懂自己有哪些模型源 | `runtime/config/cli.py` | 将 Provider 列表按“已配置 key / 缺 key / 本地 / 自定义”分组；展示 homepage、base_url 脱敏状态、模型数量 | `/models list` 不再像原始 catalog dump，普通用户能直接判断下一步该填 key 还是选模型 |
| 2.7 `/models available` 联动 | 只展示当前真的可用于运行的模型 | `runtime/config/cli.py`、`catalog_system/model_catalog.py` | 将可运行判断和 Provider 配置状态统一；对未探测模型给“未确认可用”而不是假装可用 | `/models available` 只展示配置完整且策略允许的模型 |
| 2.8 错误产品化 | 配置错时给中文修复建议 | Provider registry、OpenAI-compatible adapter、CLI 输出 | 捕获 key 错、base_url 错、model name 错、网络错、工具不支持；统一转成中文提示 | 错误不冒 Python traceback，用户知道该改 key、base_url 还是模型名 |
| 2.9 与调音台联动 | Provider 配好后能立刻进入模型调音台选择 | `runtime/config/model_tuner.py`、`runtime/commands/completion.py` | connect 后刷新模型 catalog；`/models` 可选项展示新 provider；补全缓存失效 | `connect -> /models -> select` 不需要重启也能看到新模型 |
| 2.10 回归与真实验收 | 保证第二阶段不破坏第一阶段和执行链 | `tests/run_regression.py`、真机 conda 环境 | 增加 Provider 配置、auth 脱敏、自定义中转、调音台联动测试；跑 `lucode doctor/connect/models/chat` | 定向测试通过，全量回归通过；真机可用 DeepSeek 或自定义 provider 完成一次 `/models` 选择 |

## 3. 模型能力探测

在 `/connect` 或 `/models select` 后立即探测模型能力。

探测项：

```text
chat_probe        基础聊天
json_probe        JSON 输出
tool_probe        tools/function calling
stream_probe      流式输出
context_probe     上下文长度

```

展示示例：

```text
正在检测 dashscope/qwen-max...
```

探测结果应缓存为模型能力指纹，避免每次重复探测。

## 5. CLI Safety Skill + 命令风险引擎

新增：

```text
skills/cli-command-safety/SKILL.md
```

模型生成命令前必须遵守：

- 优先只读命令。
- 禁止裸删除、强制重置、远程脚本执行。
- 命令必须说明目的、影响范围、风险、回滚方案。
- 写入、安装、发布必须审批。

系统硬规则：

```text
CommandRiskAnalyzer v2
CommandExecutionPolicy
```

决策类型：

```text
allow
allow_limited
ask
sandbox_preview
deny
```

直接拒绝的高危命令：

```text
rm -rf *
rm -f *
Remove-Item -Recurse -Force *
del /s /q *
git reset --hard
git clean -fdx
curl ... | sh
Invoke-WebRequest ... | iex
npm publish
twine upload
git push --force
```

## 7. 上下文管理

短期先不做知识图谱，只保留未来接口。先学习 Claude 风格的实用上下文管理。

### 命令

```text
/context
/compact
/clear
/memory
```

### 上下文分层

```text
热层：最近 N 轮对话 + 当前任务状态
证据层：本轮读取文件片段 / 命令输出摘要
规则层：AGENTS.md / 项目规则 / CLI safety skill
冷层：旧 JSONL 会话，只在 resume/compact 时摘要
```

目标：

- 不把大文件和工具输出反复塞进 prompt。
- 快满时自动压缩旧内容。
- 用户能看到上下文占用情况。

## 8. 多 Agent 共享 Context Pack

解决“一个 Agent 读文件，另一个 Agent 又读一遍”的浪费。

新增运行级共享缓存：

```text
RunContextStore
- file_snapshots
- tool_outputs
- code_search_results
- summaries
- artifacts
```

文件读取后生成 artifact：

```json
{
  "artifact_id": "file:runtime/agents/factory.py@sha256",
  "path": "runtime/agents/factory.py",
  "sha256": "...",
  "summary": "负责创建执行 Agent",
  "relevant_chunks": []
}
```

后续 Agent 看到：

```text
前序已读取文件：
- runtime/agents/factory.py
  摘要：...
  关键片段：...
```

只有需要更多上下文时才再次读取。文件 sha256 变了则缓存失效。
