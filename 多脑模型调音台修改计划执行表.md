# 多脑模型调音台修改计划执行表

## 1. 目标定义

本次模块的目标不是简单新增一个 `/models` 展示页，而是把 Lucode 的多脑模型配置做成一个可查看、可切换、可持久化、可测试、可逐步扩展的“模型调音台”。

用户最终应该可以做到：

1. 查看当前所有脑位正在使用什么模型。
2. 分别切换前置优化脑、主脑规划脑、执行专家脑、汇总脑的模型。
3. 支持中文角色名和英文角色名。
4. 切换后写入当前项目 `.lucode/config.toml`。
5. 切换后当前会话立即生效。
6. solo / serial / full 三种执行模式都能明确知道自己该用哪个脑位模型。
7. 旧命令 `/models role`、`/models select` 继续可用，不破坏已有用户配置。

## 2. 当前代码现状

| 模块 | 当前状态 | 缺口 |
|---|---|---|
| `runtime/config/model_config.py` | 已有 `select_model_priority`、`select_role_model_priority`、`normalize_model_role` | 角色只支持 `query_refiner`、`orchestrator`、`final_synthesizer`，缺少 `executor` |
| `runtime/config/settings.py` | `RuntimeSettings` 已有三脑优先级 | 缺少 `executor_model_priority`，默认模型也没有覆盖执行脑 |
| `runtime/config/cli.py` | 已有 `/models`、`/models role`、`/models select` | 展示不像调音台，角色不完整，缺少中文脑位切换体验 |
| `runtime/commands/registry.py` | 已注册模型相关命令 | 命令描述仍偏旧，缺少 `/models brain` 这类更直观入口 |
| `runtime/execution/solo_runner.py` | solo 大概率仍借用 `orchestrator` 作为执行模型 | solo 应该优先走执行专家脑 `executor` |
| `runtime/execution/dynamic.py` | serial/full 根据规划任务里的 `task.model` 执行 | 需要明确执行任务默认如何继承执行专家脑，避免配置了 executor 但运行链不用 |
| 测试 | 已有配置、命令、运行相关回归测试雏形 | 需要补四脑配置、命令解析、运行选模的定向测试 |

## 3. 不做的事

为降低风险，本轮不做这些大改：

1. 不重写 Provider 配置系统。
2. 不迁移现有 `.lucode/config.toml` 的整体结构。
3. 不删除旧命令 `/models role`。
4. 不把三脑/多脑配置拆成新的数据库或 session 存储。
5. 不重写 planner，不把调音台和模型能力探测一次性绑死。

一期保持现有 `[roles]` 配置结构，只扩展角色和展示层。后续如果要做更完整的 `[models.brains]` 结构，可以单独迁移。

## 4. 验收标准

| 编号 | 验收项 | 预期 |
|---|---|---|
| A1 | 输入 `/models` | 显示四个脑位：前置优化脑、主脑规划脑、执行专家脑、汇总脑 |
| A2 | 输入 `/models brain` | 显示多脑调音台、可用命令示例、当前项目配置位置 |
| A3 | 输入 `/models brain 主脑 dashscope/qwen-max` | 写入 `roles.orchestrator`，当前会话立即生效 |
| A4 | 输入 `/models brain 执行 deepseek/deepseek-chat` | 写入 `roles.executor`，solo 执行优先使用该模型 |
| A5 | 输入 `/models role executor openai/gpt-4.1-mini` | 旧命令兼容，效果等同 `/models brain executor ...` |
| A6 | 输入 `/models select provider/model fallback/model` | 默认模型优先级仍可用，并能作为四脑缺省 fallback |
| A7 | serial/full 模式 | 不因为新增 executor 导致规划、执行、汇总链路断裂 |
| A8 | 测试 | 定向测试、编译检查、真实 CLI 测试通过 |

## 5. 修改总策略

采用五段式微创推进：

1. 先测基线：确认当前项目状态，不带着未知失败动刀。
2. 扩数据模型：先让配置层认识“四脑”。
3. 扩 CLI 调音台：再让用户能看、能切。
4. 接运行链：最后让 solo / serial / full 真正使用配置。
5. 补测试和真机验收：每一步都有回归点。

## 7. 第二刀：扩展模型角色配置层

### 7.1 修改文件

`runtime/config/model_config.py`

### 7.2 修改点

新增统一角色定义，避免角色别名散落在多个函数中。

建议新增：

```python
MODEL_ROLES = {
    "query_refiner": {
        "label": "前置优化脑",
        "aliases": ("refiner", "query_refiner", "前置优化", "前置优化脑", "前置优化副脑"),
    },
    "orchestrator": {
        "label": "主脑规划脑",
        "aliases": ("planner", "main", "main_brain", "orchestrator", "主脑", "主脑规划", "主脑规划脑"),
    },
    "executor": {
        "label": "执行专家脑",
        "aliases": ("executor", "execution", "worker", "agent", "specialist", "solo", "执行", "执行脑", "执行专家", "专家脑", "solo_agent"),
    },
    "final_synthesizer": {
        "label": "汇总脑",
        "aliases": ("synthesizer", "final", "final_brain", "final_synthesizer", "汇总", "汇总脑", "汇总副脑"),
    },
}
```

再新增或调整：

| 函数 | 修改方式 |
|---|---|
| `normalize_model_role(role)` | 改为查 `MODEL_ROLES`，支持 `executor` 和中文别名 |
| `select_role_model_priority(...)` | 保持函数签名不变，内部自然支持 executor |
| `_dump_lucode_toml(config)` | 确认 `[roles] executor = [...]` 可以正常写出 |
| 新增 `model_role_label(role)` | 供 CLI 展示中文脑位 |
| 新增 `iter_model_roles()` | 供 CLI 和 settings 按固定顺序渲染四脑 |

### 7.3 副作用评估

风险点：

1. 改 `normalize_model_role` 可能影响现有 `/models role 主脑 ...`。
2. 角色错误提示如果改坏，会让用户不知道能填什么。

控制方式：

1. 保留原有别名。
2. 错误提示列出四个 canonical role 和中文名称。
3. 不改变 `.lucode/config.toml` 的 `[roles]` 结构。

### 7.4 定向测试

新增或更新测试：

```python
assert normalize_model_role("主脑") == "orchestrator"
assert normalize_model_role("执行") == "executor"
assert normalize_model_role("汇总脑") == "final_synthesizer"
```

## 8. 第三刀：扩展 RuntimeSettings

### 8.1 修改文件

`runtime/config/settings.py`

### 8.2 修改点

在 `RuntimeSettings` 中新增：

```python
executor_model_priority: Tuple[str, ...]
```

建议默认值：

```python
executor_model_priority=()
```

然后在加载流程中补齐。

| 位置 | 修改内容 |
|---|---|
| `RuntimeSettings` dataclass | 新增 `executor_model_priority` |
| `model_priority_for(role)` | 支持 `executor` 和中文别名 |
| `_apply_lucode_config_overrides(settings)` | 读取 `roles.executor` |
| `[model]` 默认优先级覆盖 | 当 `roles.executor` 未设置时，让默认 `[model]` 同样覆盖 executor |
| `_dynamic_default_priorities()` | 返回四个角色 |
| `_eligible_models_for_role()` | 支持 executor |
| `_role_score()` | 给 executor 增加执行模型评分逻辑 |
| `summary_zh()` 或相关展示函数 | 增加执行专家脑显示 |

### 8.3 executor 默认评分建议

执行专家脑应优先选择：

1. 支持工具调用的模型。
2. 上下文长度足够的模型。
3. 成本和速度适中的模型。
4. 不一定要最强规划能力，但要稳定执行文件读取、编辑、终端任务。

如果当前模型注册信息没有细粒度能力字段，先采用保守规则：

1. 优先已配置且可用的 provider 模型。
2. 如果模型 metadata 标记支持 tools，排在前面。
3. 否则保持用户配置顺序，不强行猜测。

### 8.4 副作用评估

风险点：

1. `RuntimeSettings` 构造参数增加字段，可能影响测试里的手动实例化。
2. 默认模型同步到四脑可能改变部分模式的模型选择。

控制方式：

1. 给 `executor_model_priority` 提供默认值。
2. 只在角色未显式配置时使用 `[model]` 默认值。
3. 显式配置优先级最高。

### 8.5 定向测试

新增测试场景：

| 场景 | 预期 |
|---|---|
| `.lucode/config.toml` 只有 `[model] primary = "deepseek/deepseek-chat"` | 四脑都能得到默认模型 |
| `.lucode/config.toml` 同时有 `[model]` 和 `roles.executor` | executor 使用自己的显式配置 |
| `settings.model_priority_for("执行")` | 返回 `executor_model_priority` |

## 10. 第五刀：接入 solo 执行链

### 10.1 修改文件

`runtime/execution/solo_runner.py`

### 10.2 修改点

找到 solo 创建 agent 或选择模型的位置。

当前如果类似：

```python
model_id = settings.select_model_id(model_registry, "orchestrator")
```

改为：

```python
model_id = settings.select_model_id(model_registry, "executor")
```

如果当前函数没有直接暴露 role，而是通过 `settings.orchestrator_model_priority` 获取，则改成 `settings.executor_model_priority`。

### 10.3 预期

solo 模式是执行任务，不是规划任务，因此应默认使用执行专家脑。

### 10.4 副作用评估

风险点：

1. 用户没有配置 executor 时，solo 不能断。
2. executor 配置了不存在的模型时，错误提示要清晰。

控制方式：

1. `executor_model_priority` 未配置时继承 `[model]` 默认优先级。
2. 如果 `[model]` 也没有，继续走现有动态默认选择。
3. 不在 `solo_runner.py` 自己写 fallback 逻辑，统一交给 settings/model registry。

### 10.5 测试

新增或更新 solo 测试：

| 输入配置 | 预期 |
|---|---|
| `roles.executor = ["deepseek/deepseek-chat"]` | solo 选择 deepseek |
| `roles.orchestrator = ["dashscope/qwen-max"]`，`roles.executor = ["deepseek/deepseek-chat"]` | solo 不使用 orchestrator |
| 只配置 `[model] primary` | solo 正常继承默认模型 |

## 11. 第六刀：接入 serial/full 执行链

### 11.1 修改文件

优先检查这些文件：

1. `runtime/execution/dynamic.py`
2. `runtime/execution/task_runner.py`
3. `runtime/agents/factory.py`
4. `planning/planner_schema.py`

### 11.2 设计原则

serial/full 里有两种模型选择：

1. 主脑规划脑：用于生成计划。
2. 执行专家脑：用于执行每个任务。
3. 汇总脑：用于最后总结。

不能让 executor 配置只停留在展示层。

### 11.3 推荐第一阶段策略

新增一个小 helper，不重写 planner：

```python
def apply_executor_model_defaults(plan, settings, model_registry):
    ...
```

放置位置优先级：

1. 如果已有 plan normalization 模块，放到该模块。
2. 如果没有，先放在 `runtime/execution/dynamic.py` 附近，后续再迁。

调用位置：

1. planner 生成 plan 之后。
2. plan validate 之前或 validate 之后都可，但推荐 validate 之前，使最终校验能检查 executor 模型是否合法。

### 11.4 helper 行为

建议一期采用保守逻辑：

1. 如果 task.model 为空，填入 executor 默认模型。
2. 如果 task.model 不在当前已配置模型里，且 executor 有可用模型，替换为 executor 默认模型。
3. 如果 task.model 是 planner 明确选择的可用模型，暂不强行覆盖。
4. 如果任务需要 MCP/tools，而 executor 模型不支持 tools，保留原模型并输出/记录中文 warning。

### 11.5 为什么不一刀全部强制覆盖

planner 可能根据 skill、任务类型、隐私模式选择特定模型。如果强行把所有 task.model 覆盖成 executor，可能导致：

1. 某些 skill 指定模型失效。
2. 隐私模式下错误使用云模型。
3. 多 Agent 编排失去模型差异化能力。

所以一期先做“默认填充和非法修正”，后续再做“强制执行脑模式”开关。

### 11.6 后续可加开关

后续可在 `.lucode/config.toml` 增加：

```toml
[model_tuning]
force_executor_for_tasks = false
```

但本轮先不加，避免配置面变大。

### 11.7 测试

| 场景 | 预期 |
|---|---|
| planner task.model 为空 | 自动填 executor |
| planner task.model 是不存在模型 | 自动回落 executor |
| planner task.model 是有效模型 | 保持不变 |
| executor 模型不存在 | 不吞错，给中文错误 |
| serial 三任务 | 每个执行任务都有明确模型 |
| full 并行三任务 | 每个并行任务都有明确模型且汇总仍用 final_synthesizer |

## 14. 回滚和保护策略

| 风险 | 保护方式 |
|---|---|
| 新角色导致旧配置无法读取 | 保留 `[roles]` 结构和旧三脑别名 |
| `/models` 输出变化破坏用户习惯 | `/models role`、`/models select` 完整保留 |
| executor 未配置导致 solo 失败 | executor 默认继承 `[model]`，再走动态默认 |
| serial/full 模型被错误覆盖 | 一期只填空值和修正非法模型，不强制覆盖有效模型 |
| Provider 未配置 | 返回中文提示，不直接 traceback |
| prompt_toolkit 菜单受影响 | 命令注册表只增不删，旧命令仍在 |

## 15. 推荐提交顺序
