# Java 开发规范

## 适用场景

本规范适用于 Java / Spring Boot / MyBatis-Plus 项目。当检测到 `.java` 文件或用户显式要求 Java 开发时，按本规范的强制性规则执行。

## 强制性规则

### 接口文档

- 为新增或修改接口补齐 Knife4j/OpenAPI 文档信息。
- 保证接口文档至少覆盖请求示例(不用填内容)、请求参数、响应状态、响应参数、响应示例，要和数据库字段一一对应。
- 若项目已有统一注解封装或文档风格，优先沿用现有实现，不要平行再造一套。

### 对象创建与属性映射

- 优先使用 Hutool 的 `BeanUtil.toBean`、`BeanUtil.copyProperties` 或项目已有同类封装完成对象转换。
- 优先使用 Builder 模式创建 DTO、VO、Entity、返回对象，避免长串 `setXxx()`。
- 只有在现有类型不支持 Builder 或转换工具不适用时，才退回显式赋值。

### 查询与更新写法

- 优先使用 `lambdaQuery`、`lambdaUpdate`、`LambdaQueryWrapper`、`LambdaUpdateWrapper` 或项目内等价的 lambda 风格 API。
- 避免基于字符串字段名拼接查询条件。
- 优先复用现有 service/mapper 封装，不要为了满足规范硬拆现有稳定模式。

### 空值判断与工具类使用

- 优先使用 `ObjectUtil.isEmpty()`、`ObjectUtil.isNotEmpty()` 等工具方法处理空值、空串、空集合判断。
- 没有必要时不要写零散的 `== null` / `!= null` 判断链。
- 若项目已有更统一的判空工具并且团队长期使用，优先保持仓库一致性。

### 配置约束

- 能直接在 `application.yml` 或 `application.properties` 中配置的内容，不要额外写 `config` 类。
- 只有当配置必须通过 Java Bean、条件装配或复杂初始化表达时，才新增配置类。

### 参数校验

- 把基础参数校验放在 DTO 层，优先使用 `jakarta.validation` / `javax.validation` 注解。
- 在 Controller 入参处配合 `@Valid` 或 `@Validated` 触发校验。
- 不要把通用字段合法性校验下沉到业务层，避免污染 service。
- 只有业务规则校验、跨字段校验、权限校验等确实属于业务层的内容，才放在 service 中。

### 数据库访问

- 禁止在循环中多次查询数据库形成 N+1 访问。
- 需要关联数据时，先收集 ID，再批量查询，一次组装结果。
- 不要手动设置数据库自动维护的创建时间、更新时间字段，除非当前项目并未启用自动填充或数据库维护机制。

### 注释要求

- 为类、方法、关键字段、复杂分支、批量处理逻辑、非直观语句补充简明中文注释。
- 连续业务步骤优先写在代码行上方；单行补充说明可写行尾注释。
- 注释重点解释"为什么这样做"和"这段逻辑在业务上处理什么"，不要只机械复述代码字面含义。
- 在不破坏可读性的前提下，注释密度比普通项目略高，目标是方便后续维护者快速理解。
